The Gap

Hermes has built-in memory. So why does it still forget?

The built-in system does a lot right. But in real-world use, three fundamental gaps remain — gaps this skill was built to close, and nothing else.

📠

Tight Token Budget

Built-in memory caps at roughly 800 tokens. That's about one conversation's worth of key facts. Start a second session and earlier context has already been squeezed out.

🔄

Context Vanishes on Restart

Every fresh /new or /reset is a blank slate. You have to re-explain who you are, what you were working on, and what was decided. Session search helps — but it's manual, slow, and unstructured.

📖

No Human-Agent Shared Space

Today's memory is invisible — it lives in opaque vectors and compression artifacts. You can't read it, edit it, or correct it before it's used. Memory happens to you, not with you.

Why This Works

Not a replacement. A supplement.

It doesn't touch, override, or duplicate Hermes' built-in memory. It lives beside it — in plain Markdown files your agent can read and you can review.

1

Super-Lightweight by Design

No vector database. No embedding pipeline. No cloud service. No background daemon. The entire system is a handful of Python scripts and Markdown files that sit in your Obsidian vault. Less than 26 KB downloaded. Zero install friction.

2

Shared Memory — Human + Agent

Every captured session is a Markdown file you can open, read, and edit. Spot a mistake in what was recorded? Fix it before the nightly summary runs. Want to add a note? Write it in. The memory is yours — not a black box. This is the single biggest improvement over opaque vector stores.

3

Powered by Hooks — Zero Agent Tax

Instead of asking Hermes to split its attention between answering your question and remembering to save the conversation, this uses Hermes' plugin hook system. Hooks fire automatically on each turn — pre_llm_call captures your prompt, post_llm_call captures the response. The agent never thinks about memory. It just works. This is the same pattern used by production-grade observability and logging systems — fire-and-forget observers that can never crash the main process.

4

Human-in-the-Loop Compression

The nightly summary doesn't run on raw prompts alone. You can review the day's session, add context, correct errors, or flag important decisions — and only then does the cron job compress it. This turns memory from a passive log into an active collaboration between you and your agent.

Features

Everything you need. Nothing you don't.

Purpose-built to fix specific gaps — not a kitchen-sink knowledge management platform.

Auto-Capture

Every turn, automatically. Plugin hooks log your prompt and Hermes' response to the daily session file. No manual effort. No forgetting to save.

Nightly Summaries

A cron job runs after midnight, reads the day's session, and compresses only Hermes' responses into a structured summary — decisions, fixes, errors, pending tasks.

Task Cross-Check

Every nightly summary automatically compares newly identified pending tasks against your existing Todo Index. Nothing slips through — and duplicates get caught.

Context Recovery

Start any session and the plugin automatically injects the last 3 nightly summaries into context. Hermes picks up exactly where you left off — without re-reading raw logs.

Privacy First

Zero external dependencies. No data leaves your Obsidian vault. No telemetry. No cloud. Everything runs locally on your machine or Hermes' terminal container.

Test-Driven & Verified

Built with TDD: every component has a test spec. The full suite is 42 tests and zero failures. Not because we got lucky — because each script was tested before the next one was written.
